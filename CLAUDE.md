# pveCloud — Claude Code 协作入口

本文件定义 Claude Code 在本仓库中的工作方法。不承载接口字段、表结构、页面契约或产品说明。

## 双轨协作原则

- **AI 轨道**：`CLAUDE.md` 和 `.claude/` 和 `AGENTS.md` — 约束 AI 的工作流、守则和验收方式
- **项目轨道**：`docs/`、`server/migrations/`、`server/config.example.yaml` — 承载事实、契约、架构、计划和进度

不要把两条轨道混写。

## 权威来源

- 接口契约：`docs/server/api/`
- 后端架构与业务规则：`docs/server/`
- 管理端前端契约：`docs/admin/`
- 用户端前端契约：`docs/web/`
- 数据库可执行契约：`server/migrations/`
- 配置示例契约：`server/config.example.yaml`

如果本文件和项目文档冲突，以项目文档、迁移 SQL 和配置示例为准。

对于非纯 `admin` 视觉/UI 任务，先按联动开发判断：

- `web`: 用户侧路由、状态、请求封装、登录流、展示行为
- `admin`: 运营页面、菜单、权限和管理流程
- `server/api`: `/api/*` 和/或 `/admin-api/*` 契约、鉴权、校验、响应结构
- `database`: 迁移、索引、约束、事务边界、持久状态
- `operations`: 配置示例、本地启动、部署代理边界、运行依赖

默认规则：只要不是明确的纯 `admin` 视觉/UI 调整，就按联动功能处理，先补齐受影响面的文档/契约，再进入实现。

## 必读顺序

1. `CLAUDE.md`（本文件）
2. `.claude/skills/pvecloud-document-first.md`
3. `.claude/skills/pvecloud-workflow.md`
4. 对应领域的项目文档或机器契约（见下方）

按任务类型继续读取：

- 后端与 API：`docs/server/`、`docs/server/api/`
- 管理端前端：`docs/admin/`
- 用户端前端：`docs/web/`
- 数据库：`docs/server/database/`、`server/migrations/`
- 配置与部署：`docs/development/`、`docs/operations/`、`server/config.example.yaml`
- 计划与进度：`docs/analysis/`、`docs/plan/`、`docs/progress/`

## 文档先行门禁

契约或行为变更必须先更新文档或机器契约，再暂停等待确认：

- API、错误码、鉴权、权限码、菜单来源、路由语义
- 数据库结构、迁移、事务边界、幂等规则
- 页面行为、页面范围、状态语义、请求包装、权限判断
- 配置项、部署方式、运行依赖、运维流程
- 业务流程、阶段边界、产品开放范围

联动功能必须遵守：

- 用户可见业务功能默认需要 `web + server/api + database`
- 运营或管理功能默认需要 `admin + server/api + database`
- 同一业务事实同时被用户和运营端使用时，默认需要 `web + admin + server/api + database`
- 涉及配置、启动、代理或依赖顺序时，必须联动 `operations`
- 不允许先做一个端、再口头约定后补另一个端
- 不允许先写实现、后补契约

纯 UI/UX 视觉优化可直接实现，不触发确认门禁：

- 布局、间距、颜色、字体、图标、响应式样式
- 不影响流程和契约的静态文案调整

## 标准工作流

1. `git status --short`，确认工作区是否有未提交改动
2. 读取本文件、`.claude/skills/pvecloud-document-first.md`、`.claude/skills/pvecloud-workflow.md` 和对应项目文档
3. 如果刚拉取代码、工作区已有改动、或用户说继续未完成任务 → 先做漂移检查（见下方）
4. 判断是契约/行为变更、纯视觉优化、AI workflow，还是仍需方案澄清
5. 契约/行为变更 → 先更新文档或机器契约 → 暂停等待确认
6. AI workflow 变更 → 直接改 AI 工作流文件，但不能写入或覆盖项目契约
7. 纯视觉优化 → 直接实现
8. 只在与任务直接相关的目录内改动，避免顺手扩散
9. 用最小但有意义的验证命令收尾
10. 报告改动内容、验证结果、残留风险和必要的后续建议

## 漂移检查

文档先行也约束"代码已经先变了"的接手场景。触发条件：

- 刚执行过 `git pull`
- `git status` 显示已有用户或上游改动
- 用户要求"继续未完成任务"
- 新功能代码已出现，但文档仍停留在旧阶段、旧口径

检查方法：

- 用 `git status`、`git diff`、`git log` 或定向读文件确认代码实际变化
- 把变化映射到权威项目文档或机器契约
- 同步检查 `docs/analysis/`、`docs/plan/`、`docs/progress/`，避免计划和进度仍描述旧范围
- 如果同一事实在多个文档里说法不一致，先按权威来源顺序判断，再修正文档口径
- 如果代码和文档已不匹配，先回填或修正文档/契约，暂停请维护者确认，再继续实现

## 定向读取

项目变大后，不要把整个 `docs/` 一次性读进上下文：

- 如果目录下有 `README.md`，先读这个索引
- 再按任务只读对应页面、接口、数据库、配置或进度文档
- 改 admin 页面时，先读 `docs/admin/README.md`，再读相关页面和路由文档
- 改 API 时，只读相关接口文档和必要的 conventions
- 改数据库时，只读相关设计段落和迁移文件
- 改进度或阶段边界时，才读 `docs/progress/` 的对应文件
- 如果一次任务需要读取太多不相关文档，先拆任务或暂停确认范围

## 进度文档使用规则

`docs/progress/` 是阶段账本，用来恢复背景、验收状态和范围变化历史：

- 读进度文档时，先把它视为历史/状态资料，而不是最终契约
- 写完功能或阶段后，长期有效事实必须进入权威文档（`docs/admin/`、`docs/server/api/`、迁移、配置示例等）
- 如果旧进度文件容易误导后续实现，应归档或标注历史状态
- 当进度文档和权威文档冲突时，不继续写代码；先同步文档口径并等待确认

## 仓库级硬规则

- 不覆盖用户未提交改动
- 维护者要求提交时，提交信息遵循 Conventional Commits 1.0.0：`<type>[optional scope][!]: <description>`；说明内容默认使用中文，非平凡变更必须在 body 写清背景、主要变更、验证和风险
- `admin/` 和 `web/` 独立管理，不创建跨前端共享运行时代码包
- 管理端只调用 `/admin-api/*`
- 用户端只调用 `/api/*`
- `admin/` 和 `web/` 均已存在；页面范围、路由语义和业务开放状态仍以对应 owner docs 为准
- 不要把单面临时可运行当成完整交付
- 不重建已移除的页面、路由或菜单，除非文档先更新且维护者确认

## 设计与实现原则

视觉设计关键词：

- 现代化、简洁、专业、科技感、高效、清晰、一致、优雅、流畅、直观

交互设计关键词：

- 响应式、即时反馈、平滑过渡、直观操作、快速响应、智能提示、友好错误处理

用户体验关键词：

- 易用性、可访问性、可扩展性、可维护性、性能优化、用户友好、操作便捷

技术实现关键词：

- 组件化、模块化、类型安全、状态管理、数据缓存、实时更新、错误边界

协作八荣八耻：

- 以瞎猜接口为耻，以认真查询为荣
- 以模糊执行为耻，以寻求确认为荣
- 以臆想业务为耻，以人类确认为荣
- 以创造接口为耻，以复用现有为荣
- 以一遍通过为耻，以回头校验为荣
- 以破坏架构为耻，以遵循规范为荣
- 以假装理解为耻，以诚实无知为荣
- 以盲目修改为耻，以谨慎重构为荣

奥卡姆剃刀原则：如无必要，勿增实体。

- 在多个能够解释相同现象的方案中，选择假设最少、最简单的解释
- 避免引入不必要的复杂性、依赖和抽象
- 优先使用简单直接的解决方案，而非过度设计的复杂架构
- 不要为未来可能不会发生的需求预先设计复杂的扩展性
- 每增加一个组件、依赖、抽象层时，都要问：这真的必要吗？

## 代码注释策略

新增或修改非平凡代码时，默认补充高价值详细注释。
注释用于解释业务意图、约束和风险，不用于机械复述代码。

- 复杂业务流程、状态机、事务边界、幂等键、权限裁决、安全校验、外部副作用、失败恢复和跨模块联动必须写清意图。
- 复杂函数按关键步骤添加行前或块级注释，说明“为什么这样做”和“该步骤保护什么约束”。
- 前端复杂页面、组合式状态、权限按钮显隐、异步交互和错误恢复流程，应在关键状态或关键事件处理处补充注释。
- 数据库迁移、种子数据、条件唯一约束、生成列、回填和历史兼容处理，应补充 SQL 注释说明目的和重复执行语义。
- 禁止给自解释的简单赋值、普通字段映射、显而易见的 if/return 和常规 CRUD 添加机械注释。
- 维护者明确要求“逐行注释”时，只对指定文件、函数或关键流程执行逐行注释模式；不要默认把全仓代码改成每行注释。
- 修改已带注释的代码时，必须同步更新或删除过期注释，不能留下与实现不一致的说明。

## 功能收尾建议

完成非平凡功能、修复或阶段验收后，最终回复默认补充一小段“后续建议”。

- 建议必须和“已完成内容”分开，不能把建议说成已实现。
- 建议优先覆盖更合适的技术路线、可复用 SDK/成熟库、测试补强、观测性、安全、性能和维护性。
- 每次给 1 到 3 条高价值、可执行建议；没有实质建议时明确写“暂无额外建议”或省略。
- 用户明确要求只给结果、不要建议或保持极简时，可以跳过该段。

## 领域守则

### 后端

- API 契约来自 `docs/server/api/`，不要只改 handler 或 DTO
- 表结构契约最终来自 `server/migrations/`
- 配置项契约最终来自 `server/config.example.yaml`
- 服务负责业务规则，handler 负责请求解析、权限声明和响应
- 目录按技术层分为 `delivery`、`usecase`、`domain`、`repository`、`integration`、`platform`、`shared`，每层下再按领域子包分组
- 管理端接口由 `server/internal/delivery/http/admin/*` 聚合路由，用户端接口由 `server/internal/delivery/http/web/*` 聚合路由
- 管理端业务编排落在 `server/internal/usecase/admin/*`，用户端业务编排落在 `server/internal/usecase/web/*`，领域规则落在 `server/internal/domain/*`
- 外部系统协议适配放在 `integration/`，业务裁决放在 usecase 或 domain
- 不把 RBAC 最终授权逻辑下放到前端
- 不把长耗时外部调用放进长事务
- 幂等必须依赖业务唯一键、状态检查或任务键，不能只依赖前端防重复点击
- 使用明确业务域命名，避免 `common`、`helper`、`manager`、`base` 等泛名
- 审计域统一使用 `audit` 命名
- 验证：`cd server && gofmt -w . && go test ./...`

### 数据库

- 最终表结构以 `server/migrations/` 为准
- 新增表、字段、索引、约束时，迁移和设计文档一起更新
- 迁移必须可重复执行，并考虑已有数据
- 不把高频查询条件塞进 JSON 字段
- 状态值由应用层常量维护，不用数据库 enum
- 事务只包本地必须原子完成的步骤
- 外部系统调用放到事务外，通过锚点、补偿或重试恢复
- MariaDB 为关键业务事实唯一源，Redis 只做短 TTL 状态、限流、缓存、短锁和辅助幂等

### 前端

- `admin/` 和 `web/` 完全独立，不跨应用导入任何代码
- 管理端使用 Naive UI 作为唯一基础 UI 框架，不引入第二套 UI 框架，不建设替代 Naive UI 的本地工具类体系
- 管理端图标统一使用 `@vicons/ionicons5`，不混用其它图标包
- 不再使用 Element Plus 及 `@element-plus/icons-vue`；新增页面/组件不得引入 `el-*` 组件或相关样式
- 请求包装按业务域拆分
- 路由元信息承担标题、图标、权限、菜单可见性等页面契约
- 全局状态只承载跨页面 concern，不为单页过度建模
- 验证：`cd admin && bun run build`

如果任务影响 `web/`，同步使用 `web` 的等价构建验证。

### 运维

- 配置项说明和默认语义以 `server/config.example.yaml` 为准
- 新增配置项时，先更新示例配置，再改代码和文档
- 不提交真实配置和密钥

## 历史 Basic Admin

- 只有任务明确提到历史 basic-admin、basic-admin 计划/进度文档，或旧阶段说明和当前 admin 页面范围漂移时，才使用 `.claude/skills/pvecloud-basic-admin.md`。
- basic-admin 历史资料只能解释范围变化，不能覆盖当前 owner docs、迁移、API 文档或配置示例。
- 不要用 Basic Admin 阶段说明直接重建页面、菜单或路由；当前页面范围和路由权限仍以 `docs/admin/` 为准。
