# 发票业务流程

本文档描述发票 v1 的业务流程、状态机、事务边界、幂等和安全口径。
发票接口字段以 `docs/server/api/invoices.md` 为准，退款联动接口以 `docs/server/api/orders-payments-wallet.md` 为准，数据库结构以 `server/migrations/041_invoice_v1.sql` 和 `docs/server/database/design.md` 为准。

## 业务范围

发票 v1 只开放用户端已支付订单的人工电子普通发票申请和管理端运营处理。

- 支持个人和企业电子普通发票。
- 支持一张发票申请合并多笔订单。
- 支持管理端线下开票后登记发票号码、开票时间并绑定 PDF 文件。
- 支持用户端查看自己的发票申请和下载已开票 PDF。

v1 不开放增值税专用发票、红冲、作废、第三方开票平台、发票邮件发送、钱包充值开票、部分退款、开票后在线退款或自动对账。

## 可开票订单

用户端可申请开票的订单必须同时满足：

- 属于当前登录用户。
- `payment_status` 为 `paid` 或 `manual_confirmed`。
- 订单状态不是 `cancelled` 或 `closed`。
- 未被退款，且不存在退款中的记录。
- 未被其它有效发票申请占用。
- 币种为 `CNY`。

有效发票申请指状态为 `pending`、`processing` 或 `issued` 的申请。`cancelled` 和 `rejected` 不继续占用订单。

钱包充值不属于发票 v1 开票来源。钱包余额支付形成的订单支付成功后，仍按订单申请发票，不按钱包充值申请。

## 状态机

发票申请状态：

- `pending`：用户已提交，等待运营处理。
- `processing`：运营已受理，线下开票处理中。
- `issued`：运营已完成开票登记并绑定 PDF 文件。
- `rejected`：运营驳回申请。
- `cancelled`：用户取消申请。

允许的状态流转：

- 用户创建申请：新建为 `pending`。
- 用户取消申请：仅允许 `pending -> cancelled`。
- 运营受理：仅允许 `pending -> processing`。
- 运营驳回：允许 `pending -> rejected` 或 `processing -> rejected`。
- 运营开票：仅允许 `processing -> issued`。

非法状态操作返回 `409xx` 状态冲突。跨用户访问按资源不存在或无权限处理，不通过错误文案泄露他人资源是否存在。

## 并发、锁和幂等

创建发票申请必须在一个数据库事务内完成：

1. 使用当前登录用户和 `client_token` 检查幂等申请。
2. 锁定选中的订单记录，校验归属、支付状态、退款状态和币种。
3. 写入发票申请主记录。
4. 写入订单明细和订单占用投影。

`invoice_application_orders` 必须通过生成列或等价持久约束保证同一订单同时最多被一条有效发票申请占用。并发重复选择同一订单时，数据库唯一约束兜底，服务端返回 `409xx`。

用户端创建申请的幂等键来自请求 `client_token`，同一用户同一 `client_token` 重复提交返回已有申请。不同订单集合或不同抬头资料复用同一 `client_token` 时，返回已有申请，不创建新事实。

运营受理、驳回、开票和后台备注更新均以当前申请状态为最终裁决；重复点击或多管理员并发时，先提交且状态流转成功的一方生效，后续请求按最新状态返回成功结果或 `409xx`。

受理、驳回、开票和用户取消申请时，服务端必须在同一事务内同步更新 `invoice_application_orders.status_snapshot`。该快照驱动有效订单占用投影；如果主申请状态和明细快照不一致，应视为数据漂移并优先修复，不得继续依赖前端展示状态判断可开票资格。

## 退款联动

支付退款接口在发起退款前必须检查目标订单是否存在有效发票申请。

- 订单存在 `pending`、`processing` 或 `issued` 发票申请时，不允许退款，返回 `409xx`。
- `cancelled` 或 `rejected` 发票申请不阻止退款。
- v1 不提供红冲或作废能力，因此已开票订单不得通过当前退款接口继续退款。

该退款拦截必须在退款本地事务创建前完成，并与退款状态校验保持同一业务裁决口径。

## 文件和下载

发票 PDF 使用现有文件附件能力保存。运营开票时必须绑定已上传的 PDF 附件，并写入 `file_attachment_references`，引用类型为 `invoice_application`。

下载约束：

- 用户端只能下载当前用户自己的 `issued` 发票 PDF。
- 管理端必须具备 `page.invoices` 或 `invoice:view` 才能查看和下载。
- 下载响应不得暴露物理存储路径。
- 文件名进入响应头前必须安全编码。
- 受保护下载响应不得被共享缓存长期保存。

## 权限和审计

用户端不引入发票权限码；所有发票接口按当前用户资源归属裁决。

管理端权限：

- `page.invoices`：发票运营页面、列表和详情读取。
- `invoice:view` 或 `invoice:*`：查看发票申请。
- `invoice:update` 或 `invoice:*`：受理申请、更新后台备注。
- `invoice:issue` 或 `invoice:*`：登记开票并绑定 PDF。
- `invoice:reject` 或 `invoice:*`：驳回申请。

管理端受理、驳回、开票和后台备注更新必须写入普通后台操作审计。审计只保存申请编号、状态、订单编号摘要、处理结果和必要原因，不保存 PDF 内容，不保存完整税号或敏感文件路径。

## 敏感信息

发票抬头、税号、开票邮箱和 PDF 均属于业务敏感信息。

- 税号可保存原文用于开票和展示，但日志、审计、前端错误上报和运行日志中必须脱敏。
- 个人抬头不要求税号；企业抬头必须填写税号。
- v1 不采集专票地址电话、开户行和银行账号。
- 发票 PDF 不得通过公开接口或站点 Logo 下载入口访问。

## 测试口径

实现时至少补充以下长期回归：

- Service：可开票订单筛选、多订单合并申请、用户幂等、重复占用、跨用户订单、未支付订单、已退款订单、非法状态流转和退款拦截。
- Repository/migration：有效申请占用唯一约束、取消和驳回释放占用、`issued` 保持占用。
- Handler：用户端资源归属、管理端权限、PDF 下载鉴权和错误码映射。
- Frontend：用户端发票申请/列表/详情，管理端发票运营页面，低权限按钮隐藏和接口错误反馈。

## 稳定性状态

发票 v1 当前按“人工电子普通发票申请和运营登记开票”稳定范围维护。稳定范围包括：

- 用户端可开票订单筛选、发票申请创建、幂等重复提交、取消待处理申请、列表、详情和已开票 PDF 下载。
- 管理端发票列表、详情、受理、驳回、开票登记、后台备注和已开票 PDF 下载。
- 数据库级订单有效占用约束、状态快照同步、退款前有效发票拦截、文件引用校验和后台审计写入。
- 用户端资源归属、管理端权限、PDF 下载鉴权、文件路径安全和统一错误码映射。

已完成的自动化稳定性验证包括：

- Service 回归：用户端发票幂等、取消释放订单占用、重复占用拦截；管理端受理、开票、文件引用和审计；支付退款遇到有效发票申请时拒绝。
- Handler 回归：用户端未登录、参数校验、跨用户 PDF 不可见、非 `issued` 下载冲突、成功下载响应头；管理端查看/下载权限、写操作权限、参数校验和缺失资源错误码。
- 全量后端回归：`PVECLOUD_TEST_MYSQL_DSN=... GOCACHE=/tmp/pvecloud-go-build go test ./...` 通过。
- 前端构建回归：`admin` 和 `web` 均执行 `bash -ic 'bun run build'` 通过。

稳定性边界：

- 发票 v1 不依赖第三方开票平台，因此不存在外部开票回调、开票平台幂等、平台异常补偿或自动邮件发送稳定性承诺。
- 专票、红冲、作废、钱包充值开票、部分退款和开票后在线退款不在稳定范围内；后续开放必须重新补契约、迁移、流程、权限、测试和验收记录。
- 生产上线前仍需确认文件存储根路径、反向代理鉴权头透传、受保护下载缓存头和管理端附件上传大小/MIME 限制与部署环境一致。
