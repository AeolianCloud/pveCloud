# 用户实名设计

本文档定义用户实名能力的当前业务口径。接口字段以 `docs/server/api/endpoints.md` 为准，表结构和后台配置种子以 `server/migrations/` 为准。

## 功能范围

实名能力用于记录用户提交的个人实名资料，并通过外部实名供应商完成核验。

当前阶段只开放个人实名，并支持以下供应商：

- `alipay`：支付宝开放平台身份认证流程。
- `wechat`：微信侧实名核验流程，当前按腾讯云人脸核身/实名核验能力接入。

当前阶段不开放企业实名、后台直接代填实名资料、后台人工审核、用户端上传身份证图片、银行卡四要素、公安联网直连核验或用户端删除实名记录。

## 供应商流程

实名提交采用“用户提交资料 -> 服务端创建本地申请 -> 服务端创建外部核验会话 -> 用户跳转供应商完成核验 -> 服务端回调或主动查询 -> 写入最终实名状态”的流程。

服务端是实名状态的最终裁决点：

- 前端不得根据支付宝、微信或腾讯云返回的 URL 参数直接判定实名通过。
- 已开放的外部回调必须验签，并按本地申请编号、供应商会话编号和当前状态幂等落库。
- 用户返回 Web 后，前端只能调用服务端同步接口触发服务端查询供应商结果。
- 外部供应商响应不得整包落库；只保存必要状态、错误码、脱敏说明、响应摘要和链路号。

### 支付宝

支付宝供应商使用开放平台身份认证能力。服务端创建核验会话时调用支付宝初始化接口，用户侧跳转支付宝认证入口完成认证，服务端通过回调和查询接口确认结果。

本地只保存支付宝返回的认证会话标识和结果摘要，不保存支付宝完整响应、证件号码明文或人脸图片。

### 微信

微信供应商使用微信侧实名核验流程，当前实现目标按腾讯云人脸核身/实名核验能力接入。服务端创建核验会话时获取供应商会话标识和跳转地址，用户在微信侧完成核验后，服务端通过查询接口确认结果；当前不开放微信/腾讯云异步回调处理。

本地只保存微信/腾讯云侧会话标识和结果摘要，不保存完整响应、证件号码明文或人脸图片。

## 后台配置

实名功能的业务开关、可用供应商、供应商接入参数、第三方密钥、回调地址和证件摘要密钥全部来自后台 `system_configs`，通过系统设置页维护，不走 `server/config.yaml` 或 `server/config.example.yaml`。

后台系统配置按中文分组“实名设置”展示，至少包含：

- `real_name.enabled`：是否开放用户端实名入口，默认 `false`。
- `real_name.required_for_order`：购买机器前是否要求实名通过，默认 `true`。
- `real_name.allowed_providers`：用户端可选实名供应商，默认 `alipay,wechat`；只允许 `alipay`、`wechat`。
- `real_name.default_provider`：默认实名供应商，默认 `alipay`。
- `real_name.identity_digest_secret`：证件号码 HMAC 摘要密钥，敏感配置；启用实名前必须设置，已有 HMAC 实名记录后不允许通过普通系统设置直接修改。
- `real_name.callback_base_url`：实名供应商回调基础地址，例如 `https://api.example.com/api/real-name/provider-callbacks`。
- `real_name.resubmit_enabled`：核验失败后是否允许用户重新提交，默认 `true`。
- `real_name.max_submit_attempts`：同一用户最大提交次数，默认 `3`。
- `real_name.review_notice`：用户端实名说明文案，默认空。

支付宝供应商配置同样保存在后台“实名设置”分组，至少包含：

- `real_name.alipay.enabled`
- `real_name.alipay.app_id`
- `real_name.alipay.gateway_url`
- `real_name.alipay.app_private_key`
- `real_name.alipay.alipay_public_key`
- `real_name.alipay.return_url`
- `real_name.alipay.notify_url`

微信侧供应商配置同样保存在后台“实名设置”分组，至少包含：

- `real_name.wechat.enabled`
- `real_name.wechat.secret_id`
- `real_name.wechat.secret_key`
- `real_name.wechat.region`
- `real_name.wechat.endpoint`
- `real_name.wechat.rule_id`
- `real_name.wechat.redirect_url`

配置约束：

- 布尔配置使用 `value_type=bool`，保存字符串 `true` 或 `false`。
- 数值配置使用 `value_type=int`，服务端按正整数校验。
- 供应商列表使用 `value_type=string`，以英文逗号分隔。
- 第三方应用密钥、私钥、SecretId、SecretKey、回调验签密钥和证件摘要密钥必须使用 `is_secret=1`；后台查询不得回显明文，只能返回是否已配置。
- 非敏感供应商开关、应用 ID、网关地址、回调地址、返回地址和规则 ID 使用 `is_secret=0`。
- `real_name.allowed_providers` 只决定用户端可选项；具体供应商还必须满足对应 `real_name.<provider>.enabled=true` 且必要后台配置完整。
- 用户端公开配置中的 `allowed_providers` 必须由服务端过滤，只返回后台允许、已启用且必要配置完整的供应商；`default_provider` 必须落在过滤后的可用供应商集合内。
- 公开站点配置接口只返回实名公开配置白名单，不得因为配置键以 `real_name.` 开头就暴露供应商密钥或后台接入参数。

## 实名状态

用户当前实名状态由最新有效实名申请决定：

- `unverified`：未提交或无有效实名记录。
- `pending`：已创建实名申请，等待用户完成供应商核验，或等待供应商回调/查询结果。
- `approved`：供应商核验通过。
- `rejected`：供应商核验失败，或用户放弃/超时后被服务端判定失败。

实名申请状态使用字符串字段，不使用数据库 enum。

状态流转：

- 用户提交后状态为 `pending`。
- 供应商核验通过后状态为 `approved`。
- 供应商明确失败、用户取消或核验超时后状态为 `rejected`，应保存用户可见的失败原因。
- `pending` 状态不允许用户重复提交或重复创建供应商会话；用户返回页面后只能触发同步查询。
- `approved` 状态不允许用户端自行覆盖；如需重置必须由后台后续另行开放契约。
- `rejected` 是否可重新提交由 `real_name.resubmit_enabled` 和 `real_name.max_submit_attempts` 决定。
- 已达到提交次数上限、已通过、待核验或实名入口关闭时，用户端不得创建新的供应商核验会话。

## 资料与敏感数据

实名资料包括：

- 真实姓名。
- 证件类型，当前仅 `id_card`。
- 证件号码。
- 核验供应商：`alipay` 或 `wechat`。
- 外部供应商会话标识、供应商状态、供应商结果码、供应商结果说明、开始时间、完成时间、响应摘要和链路号。

存储规则：

- 证件号码不保存明文；数据库只保存带服务端密钥的查询摘要和脱敏展示值，接口只返回脱敏号码。
- 证件摘要必须使用后台敏感配置 `real_name.identity_digest_secret` 参与计算，不得使用无盐 SHA-256。
- 新增实名申请的证件摘要版本必须标记为当前 HMAC 版本；历史 `sha256-legacy` 摘要只允许用于兼容旧已通过记录的重复校验，不得作为新记录的保护方式。
- 外部供应商原始响应、证件照片、人脸照片、token、签名密钥和证件号码明文不得落库、写入日志或进入审计详情。
- 供应商响应只保存结果码、脱敏说明、必要链路标识和响应摘要。
- 普通用户端接口不得返回供应商完整响应或后台敏感信息。

## 图片上传

当前实名流程不要求也不允许用户上传身份证图片。原人工实名上传接口不再属于当前 API 契约。

如历史环境已经存在旧人工实名附件，当前流程只允许作为历史数据保留，不提供新的用户上传、后台预览或人工审核入口；通用文件下载接口不得因为文件管理权限而绕过历史实名附件的敏感数据边界。

## 同步与审计

- 外部供应商核验通过或失败不需要管理端人工审核即可落最终状态。
- 管理端可以查看实名申请、供应商、供应商状态和用户可见失败原因。
- 管理端可执行供应商结果同步，触发服务端重新查询供应商状态。
- 供应商结果同步必须写入 `admin_audit_logs`。
- 用户提交实名不写入管理端审计日志，但应保存申请创建时间和更新时间。

## 幂等与失败处理

- 用户提交实名必须以当前用户行锁和最新申请状态防止重复创建。
- 供应商会话创建失败时，本地不得留下可误判为可继续核验的申请；如已创建申请，必须保存为可重试或失败的明确状态。
- 已开放的回调和同步查询必须按 `provider + provider_application_id` 幂等处理；当前支付宝支持回调或主动查询，微信/腾讯云只支持主动查询。
- 外部供应商返回已通过但证件摘要已被其它用户通过时，服务端必须拒绝本地通过，并把当前申请置为 `rejected` 或冲突失败状态；兼容历史数据期间，重复校验必须同时覆盖当前 HMAC 摘要和历史 `sha256-legacy` 摘要。
- 外部供应商超时、网络失败或响应无法验签时，不得把用户置为 `approved`；应保持 `pending` 并允许后续同步查询，或在达到超时策略后置为 `rejected`。

## 购买拦截

- 用户浏览产品、套餐和价格不要求实名。
- 用户购买机器或创建机器订单前，服务端必须读取 `real_name.required_for_order`。
- 当 `real_name.required_for_order=true` 时，只有实名状态为 `approved` 的用户允许继续购买。
- `unverified`、`pending`、`rejected` 状态都必须阻止购买，并返回明确错误，引导用户进入 `/user/real-name`。
- 该校验必须在服务端购买/下单接口执行，前端提示不能替代服务端裁决。

## 当前不开放

- 企业实名
- 后台直接代填实名资料
- 后台人工审核或人工兜底实名
- 用户端上传身份证图片
- 用户端删除实名记录
- 已通过实名的用户端自行修改实名资料
- 保存外部供应商完整响应、证件号码明文或人脸图片
