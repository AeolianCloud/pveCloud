# PVE Cloud Sales Platform 设计文档

**日期：** 2026-04-21  
**状态：** 已完成会话内方案确认，待书面复核

## 1. 项目目标

建设一套完整的云服务器销售系统，目标形态参考成熟商用云控制面板，但第一阶段坚持做可落地的 MVP，不做过度拆分。

第一阶段目标覆盖：

- 官网产品展示与售卖
- 用户注册、登录、找回密码
- 订单创建、支付、订单查询
- 云服务器异步开通
- 用户侧实例管理与基础操作
- 后台商品、订单、实例、任务管理

技术栈约束如下：

- 后端：Go
- 官网前台：Bun + Vue 3 SPA
- 管理后台：Bun + Vue 3 SPA
- 主库：MariaDB
- 辅助基础设施：Redis

## 2. 已确认的架构结论

以下结论已经确认：

- 前端采用两个独立工程：`web`、`admin`
- 官网前台与管理后台不做任何前端共享
- 后端对外分为 `public-api` 与 `admin-api`
- 后端第一阶段采用模块化单体，不上微服务
- 云服务器开通与资源操作采用异步任务模式
- 前台用户账号体系与后台管理员账号体系分离
- 主业务数据统一存储在 `MariaDB`
- `Redis` 仅用于缓存、验证码、登录态、限流、幂等与任务辅助
- 官网前台不做 SEO，不采用 SSR
- 数据库所有表、字段、关键状态字段都必须写中文注释

## 3. 推荐总体形态

第一阶段推荐采用：

- 两个独立前端
- 两个 API 门面
- 一个 Go 后端主工程
- 一个异步任务执行进程

这套结构既能保持前后台边界清晰，也不会过早引入微服务带来的部署、联调、日志、鉴权、调用链复杂度。

## 5. 前端架构

### 5.1 `web`

`web/` 为官网前台与用户中心，独立开发、独立构建、独立部署。

主要范围：

- 首页
- 产品列表与详情
- 注册、登录、找回密码
- 用户资料与安全中心
- 订单列表与订单详情
- 支付发起与支付状态查询
- 我的云服务器列表与详情
- 用户侧开机、关机、重启、重装、续费等操作
- 公告、站内信、通知展示

### 5.2 `admin`

`admin/` 为管理后台，独立开发、独立构建、独立部署。

主要范围：

- 管理员登录
- 控制台仪表盘
- 用户管理
- 商品与套餐管理
- 区域、线路、镜像、节点映射管理
- 订单管理
- 支付单查看与处理
- 实例管理
- 异步任务查看与重试
- 短信、邮件、站内信模板管理
- 角色、权限、审计日志

### 5.3 前端隔离原则

第一阶段不做任何前端共享，具体包括：

- 不共用组件库
- 不共用请求封装
- 不共用前端类型包
- 不共用工具包
- 不共用前端状态管理逻辑

原因：

- 官网前台和管理后台的交互模型差异大
- 过早复用会把两个端绑定在一起
- 第一阶段比起复用，更重要的是边界清晰和迭代速度

## 6. API 门面架构

系统对外暴露两套 API：

- `public-api`
- `admin-api`

它们是接入层，不是核心业务层。

### 6.1 API 门面允许承担的职责

- 路由组织
- 登录态解析
- 权限校验
- 参数校验
- DTO 转换
- 响应结构组装
- 面向前端页面的数据聚合
- 访问日志与审计入口

### 6.2 API 门面不允许承担的职责

- 商品定价规则
- 账单计算规则
- 订单状态流转规则
- 支付状态推进规则
- 实例状态判定规则
- 异步任务重试规则

这些核心规则必须落在领域模块中，而不是落在 handler/controller 中。

### 6.3 `public-api` 范围

`public-api` 仅服务 `web`。

推荐路由分组：

- `/auth`
- `/profile`
- `/products`
- `/orders`
- `/payments`
- `/instances`
- `/notices`
- `/tickets`

它以“用户能做什么”为中心组织接口，返回结构应尽量贴近用户前端页面的直接需求。

### 6.4 `admin-api` 范围

`admin-api` 仅服务 `admin`。

推荐路由分组：

- `/auth`
- `/admins`
- `/users`
- `/products`
- `/resources`
- `/orders`
- `/payments`
- `/instances`
- `/tasks`
- `/notifications`
- `/audits`

它以“后台要管理什么”为中心组织接口，强调筛选、审查、批量操作、日志追踪与权限控制。

### 6.5 API 边界硬规则

- `web` 只能调用 `public-api`
- `admin` 只能调用 `admin-api`
- 两套路由完全分离，不共用 URL
- 允许共用领域服务，不允许共用对外接口契约

## 7. 后端领域架构

第一阶段后端采用模块化单体。每个模块应尽量自包含，负责自己的模型、仓储、服务与领域边界。

### 7.1 `user`

负责：

- 前台用户注册与登录
- 找回密码
- 用户资料
- 安全设置
- 用户侧身份能力支撑

### 7.2 `adminuser`

负责：

- 管理员账号
- 管理员登录
- 角色
- 权限
- 后台菜单能力
- 管理员登录记录与操作记录

### 7.3 `catalog`

负责：

- 产品
- SKU
- 区域
- 镜像
- 线路
- 可售规则
- 商品展示属性
- 销售 SKU 与资源节点之间的业务映射

### 7.4 `order`

负责：

- 下单
- 订单查询
- 订单状态流转
- 续费订单
- 升降配订单
- 取消规则

### 7.5 `payment`

负责：

- 支付单创建
- 第三方支付渠道对接
- 回调验签
- 支付状态同步
- 退款入口与退款渠道记录

### 7.6 `billing`

负责：

- 价格计算
- 计费周期
- 折扣规则
- 续费金额
- 到期规则
- 计费快照
- 欠费处理
- 服务周期计算

### 7.7 `instance`

负责：

- 业务侧实例记录
- 实例归属
- 用户与后台可见的实例详情
- 操作可执行性判断
- 业务状态回写

### 7.8 `resource`

负责：

- 对接自研底层虚拟机 API
- 创建虚拟机
- 开机
- 关机
- 重启
- 重装
- 查询节点
- 查询实例运行状态
- 查询底层资源可用量

说明：

- `resource` 不是订单中心
- `resource` 不是计费中心
- `resource` 只负责资源动作与资源数据适配
- 订单、查询、计费逻辑仍然在你们自己的销售系统后端中实现

### 7.9 `task`

负责：

- 异步任务创建
- 执行状态流转
- 重试
- 失败记录
- 幂等控制
- 任务执行跟踪

### 7.10 `notification`

负责：

- 短信
- 邮件
- 站内信
- 通知模板
- 发送记录

### 7.11 `audit`

负责：

- 用户侧审计
- 管理员操作日志
- 关键业务事件追踪

### 7.12 `auth`

负责：

- Token 或 Session 支撑
- 中间件
- 身份解析
- 权限校验中间件

### 7.13 `common`

负责：

- 公共错误码
- 响应封装
- 配置读取
- 公共工具
- 真正跨域的枚举与常量

## 8. 三条核心边界解释

### 8.1 `public-api` / `admin-api` 只是门面

这条规则的真实含义是：

- API 层负责“接请求”
- 领域层负责“真正怎么做”

这样做的价值：

- 核心业务规则可复用
- 前端字段调整不会污染领域逻辑
- 订单、计费、实例状态逻辑不会散落在 handler 中
- 后续测试更容易落到领域模块

### 8.2 `task` 统一承接异步任务

所有会改变资源状态、耗时长、容易失败、需要重试的动作，都统一走任务中心。

第一阶段至少包括：

- 云服务器开通
- 开机
- 关机
- 重启
- 重装
- 后续新增的资源变更操作

原因：

- 底层资源操作可能耗时较长
- 同步请求容易超时
- 用户重复点击会造成重复下发
- 异步任务更容易做重试、补偿、审计与人工介入

### 8.3 `billing` 与 `payment` 必须分开

`billing` 解决的是“该收多少钱”。  
`payment` 解决的是“钱怎么收回来”。

`billing` 负责：

- 价格
- 周期
- 折扣
- 续费规则
- 到期规则

`payment` 负责：

- 渠道拉起
- 回调验签
- 支付状态同步
- 退款记录
- 支付通道流水

这两块拆开后，后续对账、退款、0 元订单、余额抵扣、补差价等场景会清晰很多。

## 9. 实例开通与操作流程

### 9.1 开通流程

建议第一阶段按以下流程执行：

1. 用户在 `web` 下单
2. `public-api` 做鉴权与参数校验
3. `order` 创建订单
4. `billing` 计算应付金额
5. `payment` 创建支付单
6. 支付成功
7. 订单进入 `paid` 或 `provisioning`
8. `task` 创建 `create_instance` 异步任务
9. `worker` 消费任务
10. `resource` 调用底层虚拟机 API
11. `instance` 回写业务实例数据
12. 订单进入已生效状态
13. `notification` 发送通知

### 9.2 开通失败处理

如果开通失败：

- 记录底层失败原因
- 记录任务执行日志
- 任务进入失败或重试状态
- 订单进入失败或待人工处理状态
- 后台可查看并决定重试或人工介入

### 9.3 实例操作流程

对于开机、关机、重启、重装等操作：

1. 用户或管理员发起操作
2. `instance` 判断当前状态是否允许执行
3. `task` 创建异步操作任务
4. `worker` 通过 `resource` 执行动作
5. `instance` 回写结果
6. `audit` 与 `instance_actions` 记录全过程

## 10. 数据架构

### 10.1 MariaDB 与 Redis 的角色

已确认：

- 主库使用 `MariaDB`
- `Redis` 可用，但不是主业务数据存储

角色划分：

- `MariaDB` 存储所有核心业务数据
- `Redis` 仅用于：
  - 登录态
  - 验证码
  - 限流
  - 缓存
  - 幂等键
  - 异步任务辅助

### 10.2 MariaDB 设计规范

第一阶段建议统一采用：

- 存储引擎：`InnoDB`
- 字符集：`utf8mb4`
- 表名：小写下划线复数名词
- 主键：`id BIGINT UNSIGNED AUTO_INCREMENT`
- 关联字段：`*_id`
- 业务编号：`*_no`
- 金额字段：整数，单位分
- 状态字段：优先用可读字符串，如 `VARCHAR(32)`
- 时间字段：`DATETIME(3)`
- 第一阶段不强依赖数据库物理外键，由应用层维护一致性

### 10.3 中文注释硬规范

这是数据库层面的强制规则：

- 每张表必须有中文表注释
- 每个字段必须有中文字段注释
- 每个状态字段必须写明枚举值含义
- 每个金额、带宽、流量、内存、磁盘字段必须写明单位
- 每个布尔字段必须写清 `0-否，1-是`

例如：

- `订单状态：pending_payment-待支付，paid-已支付，provisioning-开通中，active-已生效，failed-失败，closed-已关闭`
- `订单原始金额，单位分`
- `是否启用：0-否，1-是`

不允许出现无注释表、无注释字段、无单位说明字段。

## 13. 错误处理与审计原则

第一阶段就要把可追踪性和可恢复性纳入设计。

要求：

- 所有关键写操作都应产生审计记录或操作日志
- 所有异步任务都应保留失败原因与重试次数
- 所有支付回调都必须可追踪
- 所有资源动作都必须能追溯到用户、管理员或系统任务
- 用户侧接口不直接暴露下游原始错误

## 17. 最终建议

第一阶段最终建议为：

- `web/` 和 `admin/` 保持完全独立
- `server/` 采用 Go 模块化单体
- 对外拆分 `public-api` 和 `admin-api`
- 使用独立 `worker` 处理异步任务
- 使用 `MariaDB` 作为主业务存储
- 使用 `Redis` 做非主存储辅助能力
- 通过统一任务中心处理开通与实例操作
- 严格执行数据库中文注释规范

这套方案能在第一阶段保持清晰边界、可维护性和扩展余地，同时避免过早承担微服务复杂度。
