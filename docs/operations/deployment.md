# 部署与运维说明

## 部署组成

```text
api
admin
web
MariaDB
Redis
```

## 代理边界

- 代理 `/admin-api/*` 到管理端 API
- 未来开放用户端 API 时，代理 `/api/*` 到用户端 API 边界
- `admin` 可使用独立域名或同域不同路径
- `web` 可使用独立域名或站点根路径
- `web` 不代理 `/admin-api/*`，`admin` 不代理 `/api/*`

## 配置与依赖

- 配置示例契约：`server/config.example.yaml`
- 真实部署配置不得提交到仓库
- Redis 是运行时基础依赖，不提供生产降级模式
- 管理端会话最终状态、RBAC 和审计最终状态仍以 MariaDB 为准
- Web 基础前台阶段只发布用户端静态前端，不代表用户端账号、产品、订单、支付、实例或工单 API 已开放

## 本地开发脚本与生产的区别

仓库根目录的 `scripts/dev.mjs` 仅面向开发环境。它不是生产进程管理方案。

## 运维关注点

- API 启动时必须检查 Redis 可用性
- `/healthz` 应能反映核心依赖健康状态
- 高危管理操作必须进入审计域
- `admin` 和 `web` 的静态资源、域名和代理边界必须分开配置
- 若未来开放 `/api/*`，需要同步更新 API 契约、后端实现边界和代理规则

## 备份与恢复

第一阶段至少覆盖：

- MariaDB 备份
- 配置安全备份

恢复演练至少覆盖管理员账号、角色权限、系统配置和审计日志。
