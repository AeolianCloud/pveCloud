# 订单管理 MVP 计划

## 目标

第一阶段开放订单管理 MVP，让用户端可以基于当前服务器产品目录创建和查看订单，让管理端可以查看和处理用户端订单。

本阶段只恢复订单域，不开放支付、实例、PVE 节点、资源池、库存扣减或自动开通。

## 范围

- 用户端从公开服务器产品目录选择套餐、计费周期、销售地域和系统模板后创建订单。
- 用户端查看自己的订单列表和订单详情。
- 管理端查看订单列表和订单详情。
- 管理端编辑后台备注、取消订单或关闭订单。
- 创建订单时保存产品、套餐、价格、销售地域和系统模板快照。
- 创建订单时后端重新计算金额，前端提交金额只可用于展示或确认，不作为最终事实。
- 当 `real_name.required_for_order=true` 时，创建订单前必须要求当前用户实名状态为 `approved`。

## 非范围

- 管理端创建订单。
- 支付发起、支付回调、支付状态确认或支付单。
- 实例创建、实例交付、实例状态、PVE 节点、资源池、模板 ID 或库存扣减。
- 钱包、余额、优惠券、发票、退款。
- 工单和异步任务。
- 订单状态提前预留支付或交付状态。

## 状态模型

订单状态只包含：

- `pending`：订单已创建，等待运营处理或后续人工流程。
- `cancelled`：订单已取消，通常由用户或管理员取消。
- `closed`：订单已关闭，通常由管理员处理完成或终止。

本阶段不使用 `paid`、`provisioning`、`fulfilled` 或其它支付、交付相关状态。

## 数据库契约

新增 `orders` 表保存订单最终事实。

关键规则：

- 主键使用 `BIGINT UNSIGNED AUTO_INCREMENT`。
- 对外展示使用 `order_no`，不得向用户端暴露自增 ID。
- 金额使用分为单位，不使用浮点数。
- 订单保存产品、套餐、价格、销售地域和系统模板快照。
- 订单状态使用字符串，不使用数据库 enum。
- MariaDB 是订单最终事实来源，Redis 只能用于短 TTL 幂等或辅助状态。
- 创建订单必须在事务中完成订单写入；不得在事务中调用外部支付、PVE 或其它长耗时外部系统。
- 管理端取消、关闭和备注变更必须写入后台审计。

## API 契约

用户端接口只挂载 `/api/*`：

- `POST /api/orders`：创建订单。
- `GET /api/orders`：查看当前用户订单列表。
- `GET /api/orders/{order_no}`：查看当前用户订单详情。
- `POST /api/orders/{order_no}/cancel`：当前用户取消自己的 `pending` 订单。

管理端接口只挂载 `/admin-api/*`：

- `GET /admin-api/orders`：查看订单列表。
- `GET /admin-api/orders/{order_no}`：查看订单详情。
- `PATCH /admin-api/orders/{order_no}/admin-note`：更新后台备注。
- `POST /admin-api/orders/{order_no}/cancel`：取消订单。
- `POST /admin-api/orders/{order_no}/close`：关闭订单。

## 管理端权限

新增管理端页面和权限：

- 菜单权限：`page.orders`
- 查看权限：`order:view`
- 更新权限：`order:update`
- 取消权限：`order:cancel`
- 全权限：`order:*`

管理端不提供创建订单权限和创建订单接口。

## 页面契约

用户端：

- 产品展示页开放下单入口。
- 用户中心展示订单管理入口。
- 新增订单列表页和订单详情页。
- 用户端只展示当前登录用户自己的订单。

管理端：

- 新增订单管理页面。
- 订单管理页面支持列表筛选、详情查看、后台备注、取消和关闭。
- 订单管理页面不支持创建订单。

## 实施阶段

1. 更新 owner docs 和迁移契约，并暂停确认。
2. 实现后端订单领域、仓储、用户端用例、管理端用例和 HTTP 路由。
3. 实现用户端下单入口、订单列表、订单详情和用户中心入口。
4. 实现管理端订单管理页面和权限接入。
5. 执行验证并回扫文档与实现漂移。

## 验证

- 后端：`go test ./...`
- 管理端：`bun run build`
- 用户端：`bun run build`

重点回归：

- 未登录创建订单。
- 未实名且 `real_name.required_for_order=true` 时创建订单。
- 跨用户查看订单。
- 非法产品、套餐、计费周期、销售地域或系统模板组合。
- 前端伪造金额。
- 重复提交订单。
- 低权限管理员访问订单接口。
- 非 `pending` 订单重复取消、关闭或状态冲突。

## 风险

- 当前工作区已有 Web 页面相关未提交改动，后续实现阶段必须避免覆盖这些改动。
- 当前阶段不开放支付和实例，因此订单只表示交易意向或人工处理入口，不承诺自动交付。
- 如果后续要接入支付或实例交付，必须另行补充支付、实例、幂等、回调、补偿和审计契约。
