# 基础后台完善计划

本计划只覆盖基础后台管理能力。产品套餐、订单、支付、实例、工单和用户端业务不进入本阶段。

## 阶段 0：安全日志基座

目标是完成普通审计日志和高危操作日志的两套日志基础，并把后台菜单收敛到当前阶段可用功能。

验收口径是接口文档、数据库设计、迁移、后端查询接口、前端日志页面和菜单入口保持一致；高危操作具备双写规则；`audit:sensitive_view` 控制敏感日志字段；后端测试和管理端构建通过。

## 阶段 1：审计写入和脱敏收口

目标是把审计写入从零散服务逻辑收口成明确的审计域能力。普通操作只写 `admin_audit_logs`，高危操作同时写 `admin_audit_logs` 和 `admin_risk_logs`。

需要完成集中审计写入服务、统一脱敏规则、高危动作矩阵、风险等级映射、写入失败处理和单元测试。敏感数据不得依赖前端隐藏，后端返回前必须处理。

验收口径是登录限制、验证码限制、管理员变更、角色变更、会话吊销和敏感配置更新都通过统一写入入口记录日志；无 `admin_log_service` 这类含糊命名；无密码、token、secret、验证码和敏感配置明文出现在接口响应。

## 阶段 2：管理员和角色权限

目标是补齐后台账号管理和 RBAC 管理闭环。

需要完成管理员列表、创建、详情、编辑、启用、禁用、重置密码、分配角色，完成角色列表、创建、详情、编辑、启用、禁用、分配权限，完成权限码只读分组查询。

验收口径是后端 RBAC 仍为最终权限边界；超级管理员账号和内置超级管理员角色受到保护；角色权限变更和重置密码写入高危日志；前端页面具备加载态、空状态、错误重试、提交中禁用和二次确认。

## 阶段 3：登录会话和系统设置

目标是补齐后台运行维护所需的会话控制和配置管理。

需要完成会话列表、会话筛选、吊销他人活跃会话、当前会话保护，完成系统配置分组查询、普通配置更新、敏感配置更新和敏感值不回显。

验收口径是会话吊销后 token 立即失效；当前会话不能通过列表吊销；敏感配置只显示是否已设置；修改敏感配置写入高危日志；普通配置更新写入审计日志。

## 阶段 4：Dashboard 和管理端体验收口

目标是让后台首页和基础管理页面形成可日常使用的工作台。

需要完成 Dashboard 真实基础指标、后台 403 页面、页面级权限提示、列表筛选 URL query 同步、表单错误映射、确认弹窗和后台内复用组件整理。

验收口径是 Dashboard 不展示未开放业务模块数据；无权限访问有明确页面反馈；刷新页面后登录态、权限、菜单和筛选条件表现稳定；复用代码只存在于 `admin/` 内部。

## 阶段 5：验收和发布准备

目标是把基础后台作为一个可提交、可回归、可继续扩展的阶段成果。

需要补齐后端服务测试、权限测试、前端构建验证、迁移检查、文档对账和种子数据检查。

验收口径是 `go test ./...`、`bun run build` 通过；接口文档、数据库设计、迁移、前端架构文档和实际菜单路由一致；业务模块仍未出现在后台菜单和受保护路由中。

## 依赖顺序

审计写入和脱敏收口优先于其它管理功能，因为后续管理员、角色、会话和系统设置都会写审计或高危日志。

管理员和角色权限优先于会话与系统设置，因为会话和配置页面依赖稳定的权限边界和操作者身份。

Dashboard 和体验收口放在基础功能之后，避免首页指标和页面状态跟随接口反复返工。
