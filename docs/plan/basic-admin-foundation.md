# 基础后台实施计划

本计划描述基础后台阶段的目标、顺序和验收重点。

## 计划目标

形成一个可登录、可授权、可追踪、可配置的基础后台底座。

## 计划边界

### 当前阶段纳入的后台能力

- 管理端认证与会话
- Dashboard
- RBAC
- 管理员账号能力
- 角色与权限能力
- 会话管理能力
- 系统配置能力
- 审计日志与高危日志能力

### 当前阶段不纳入的业务模块

- 产品套餐
- 订单
- 支付
- 实例
- 工单
- 用户端业务流

### 重要说明

上面是“后台能力”范围，不等于“当前管理端前端页面”范围。
当前前端页面范围仍以 `docs/admin/architecture.md` 为准，只保留 `Login`、`Dashboard` 和 `403`。

## 分阶段路线

### Phase 0：安全日志基座

建立审计日志、高危操作日志和对应文档、迁移、权限边界。

### Phase 1：审计写入与脱敏收口

把审计写入从零散调用收口成稳定域能力，明确高危双写、脱敏和风险等级。

### Phase 2：管理员与角色权限

补齐后台账号与 RBAC 能力，但不要求在当前前端继续保留这些独立页面。

### Phase 3：登录会话与系统设置

补齐会话管理、系统配置和敏感配置处理，但不要求在当前前端继续保留这些独立页面。

### Phase 4：Dashboard 与管理端体验收口

完成三页管理端、后台壳、Element Plus 入口、权限路由和最小复用模式。

### Phase 5：验收与发布准备

确保接口文档、数据库设计、迁移、前端架构、测试和构建结果一致。

## 验收口径

- API 契约与实现一致
- 数据库设计与迁移一致
- 当前管理端前端只保留 `Login`、`Dashboard`、`403`
- Dashboard 不展示未开放业务数据
- `go test ./...` 通过
- `bun run build` 通过

## 文档要求

如果后续要重新开放管理员、角色、会话、系统设置、审计日志或高危日志页面，必须先更新：

- `docs/admin/architecture.md`
- 相关 API 文档
- 必要时更新计划与进度说明

确认后再进入实现。
