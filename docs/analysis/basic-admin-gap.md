# 基础后台缺口分析

本文档记录当前基础后台阶段中，后端能力、管理端前端范围和剩余缺口之间的真实关系。

## 阶段目标

先完成一个可登录、可授权、可追踪、可配置的基础后台。

当前阶段不开收：

- 用户端 API
- 用户端账号
- 产品套餐
- 订单
- 支付
- 实例
- 工单
- 异步任务
- 用户端业务流

## 一个容易混淆的事实

当前“基础后台阶段”包含的后端能力范围，和“当前管理端前端开放页面范围”不是一回事。

### 后端仍在范围内的管理域

- auth
- dashboard
- admin users
- roles and permissions
- admin sessions
- system configs
- audit logs
- risk logs

### 当前前端实际开放范围

- `Login`
- `Dashboard`
- `System Settings`
- `403`

`System Settings` 当前承载系统配置和管理员设置；管理员设置中承载管理员账号、管理员组权限与管理员会话管理。
审计日志和高危操作日志相关接口、权限码和数据结构仍保留在后端，但前端页面、菜单和受保护路由当前仍未开放。

## 已具备的基础

- 管理端验证码登录
- JWT 与管理端会话
- `/admin-api/auth/me` 登录态恢复
- `/admin-api/auth/logout` 与 `/admin-api/auth/refresh`
- Dashboard 初始能力
- 管理端 RBAC 中间件
- 审计日志和高危操作日志基础结构
- 管理端前端 Element Plus 后台壳、路由中心化权限系统和当前开放页面收口

## 仍需关注的缺口

### 后端侧

- 审计写入必须保持集中统一，避免重新分散
- 会话有效性、权限边界、敏感字段脱敏和高危双写需要持续回归验证
- 管理员、角色、会话、配置等后台域能力需要继续保持契约与实现一致

### 前端侧

- 不能因为后端域能力存在，就自动恢复审计日志、高危日志或其它未确认页面
- 必须持续确保菜单、路由、页面和 Dashboard 指标不越界
- 复用模式只保留在 `admin/` 内部

### 文档侧

- 需要明确区分“当前前端实际范围”和“阶段后端能力范围”
- 需要明确 `docs/web/` 当前是规划文档而不是现有实现说明
- 需要明确 skill 与 docs 的职责边界

## 风险点

- 把 skill 当契约文档使用
- 把阶段范围误读成现有前端页面范围
- 把 `docs/web/` 误读成已有 `web/` 代码结构
- 因为历史代码保留而误恢复业务域
