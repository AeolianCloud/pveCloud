# 基础后台缺口分析

本文件记录当前阶段的后台管理范围和剩余缺口。当前目标是先形成可登录、可授权、可追踪、可配置的基础后台，不进入产品套餐、订单、支付、实例和工单等业务模块。

## 当前范围

基础后台只包含 Login、Dashboard、管理员、角色权限、登录会话、系统设置、审计日志、高危操作日志和 403 无权访问页。菜单、路由、权限、接口和文档都应围绕这些能力收敛。

后台安全日志分为两套。普通后台操作写入 `admin_audit_logs`。高危操作同时写入 `admin_audit_logs` 和 `admin_risk_logs`。普通操作不写高危日志。

敏感日志字段按权限分层。只有 `audit:view` 时只能看日志主信息。额外具备 `audit:sensitive_view` 时可以看到已脱敏的 `before_data`、`after_data` 和 `user_agent`。密码、token、secret、验证码和敏感配置明文不得进入接口响应。

## 已具备的基础

登录、验证码、JWT、管理端会话、退出登录、刷新 token、Dashboard 初始接口、RBAC 权限中间件、基础菜单收敛已经形成基础。

当前工作区已经补入审计日志和高危操作日志的文档、迁移、模型、查询接口、管理端页面、菜单入口和风险日志权限种子。该部分仍以工作区状态为准，最终进入实现验收前需要重新跑后端测试和管理端构建。

## 剩余缺口

审计和高危日志还需要集中写入能力。后续不应在每个服务里零散拼审计数据，而应通过明确的审计写入服务统一处理普通审计、高危双写、脱敏、风险等级、对象快照和写入失败策略。

管理员账号模块还缺完整后端服务、接口处理器、表单校验、前端列表、创建、编辑、启用、禁用、重置密码和角色分配。超级管理员保护规则需要落在后端。

角色权限模块还缺角色服务、角色接口、权限只读接口、权限分组展示、角色启停、角色授权和内置角色保护。权限码由迁移和种子维护，后台只读展示权限码，不提供新增或删除权限码入口。

登录会话模块还缺会话列表、会话筛选、吊销他人活跃会话、当前会话保护和高危日志记录。退出当前会话仍走 `POST /admin-api/auth/logout`。

系统设置模块还缺配置分组查询、配置更新、敏感配置写入和敏感配置不回显。修改敏感配置属于高危操作。

Dashboard 还需要从真实基础后台数据生成指标。当前阶段只统计基础后台相关指标，不展示订单、支付、实例等业务数据。

管理端前端还需要沉淀后台内可复用的列表、筛选、表单、确认、错误态、空状态和提交态模式，但只能在 `admin/` 内部复用，不创建跨 `admin/` 和 `web/` 的 shared 包。

测试缺口集中在权限边界、会话有效性、敏感字段脱敏、高危双写、管理员保护、角色权限更新和系统配置敏感值处理。

## 命名风险

后端和前端都必须避免用 `log`、`manager`、`common`、`helper`、`utils`、`data`、`base` 这类泛名承载具体业务。当前包结构较宽，导出的类型和文件名必须带业务域和职责，例如 `AdminAuditService`、`AdminUserHandler`、`SystemConfigRequest`。

审计域统一使用 `audit` 命名。高危操作日志属于审计域，不另起含糊的 `admin_log_service` 或 `risk_service` 命名。
