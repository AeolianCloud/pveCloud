# AI 协作说明

本文件面向维护者说明 `pveCloud` 中 AI 协作系统的职责划分。

它不是接口、数据库、前端或产品契约文档。

## 我们为什么要分成两条轨道

AI 需要稳定的读取顺序、门禁和实现守则。
项目本身又需要稳定的事实、契约、架构、计划和进度文档。

如果把这两类内容混在一起，会出现三个问题：

1. 文档会越来越像提示词，维护者难以把它当项目事实读取。
2. skill 会偷偷携带契约，导致真实契约分裂。
3. AI 会因为同一事实在多个地方出现而产生错读。

所以我们采用双轨协作：

- AI 轨道：`AGENTS.md` + `.codex/skills/`
- 项目轨道：`docs/` + `server/migrations/` + `server/config.example.yaml`

## AI 轨道负责什么

- 仓库入口和读取顺序
- 文档先行门禁
- 分领域实现守则
- 不可绕过的协作边界
- 最小验证与收尾习惯

## 项目轨道负责什么

- API 契约
- 数据库设计和迁移
- 前后端架构与页面行为
- 配置、部署和本地开发说明
- 阶段计划、缺口分析和进度跟踪

## 设计原则

- skill 写“怎么做”
- 文档写“系统是什么”
- 迁移和配置示例写“机器最终执行什么”

## 当前特殊说明

- `admin/` 是当前已存在的前端实现。
- `docs/admin/architecture.md` 是管理端当前架构与页面范围文档。
- `web/` 已存在，是当前实际用户端前端实现。
- `docs/web/architecture.md` 当前描述用户端前端契约；当前已开放公开站点配置、用户账号自助、用户实名和服务器产品目录展示。
- 当前仍不开放订单、支付、实例、工单、PVE 节点、资源池、库存扣减或自动开通能力。

## 维护建议

- 当 AI 频繁犯的是“工作方法错误”，改 skill。
- 当人和 AI 都会困惑的是“系统事实或契约不清”，改文档。
- 当真实执行结构变了，改迁移或 `config.example.yaml`。
