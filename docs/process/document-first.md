# 文档先行流程

pveCloud 的开发顺序是：先写文档，确认没有问题，再写代码。

## 为什么这样做

这个项目包含订单、支付、余额、实例开通、PVE、后台权限和审计。很多错误不是代码语法错误，而是边界和状态没说清楚。文档先行可以把这些问题提前暴露出来。

## 标准流程

```text
需求或想法
  ↓
找到对应工程域文档
  ↓
补齐设计、状态、接口、表结构、验收点
  ↓
人工或 AI review 文档
  ↓
确认无问题
  ↓
实现代码和迁移
  ↓
测试和复查
```

## 什么时候必须先改文档

- 新增或修改数据库表、字段、索引、约束。
- 新增或修改 API 路由、请求、响应、错误码。
- 新增业务状态或状态流转。
- 新增支付、PVE、通知等外部集成。
- 新增前端页面、路由、API 类型或状态枚举。
- 修改鉴权、权限、审计规则。
- 修改部署、配置、备份或运维流程。

## 文档验收清单

写代码前至少确认：

- 文档路径是否对应未来代码目录。
- 是否说明了谁调用、谁拥有业务规则、谁只做适配。
- 是否列出状态枚举和允许流转。
- 是否说明事务边界和幂等策略。
- 是否说明失败、重试、补偿和人工处理方式。
- 是否说明权限和审计要求。
- 是否同步了数据库文档和迁移 SQL。
- 前端变更是否分别落在 `docs/web/` 或 `docs/admin/`，没有引入公共前端包。

## AI 执行规则

AI 接到实现任务时，如果发现对应文档缺失或过时，先补文档并说明原因。只有当文档足够指导实现时，才开始写代码。

如果用户明确要求“先实现”，AI 也应检查是否已有可依赖文档；没有文档时先创建最小设计说明，再进入实现。
