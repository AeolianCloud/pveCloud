# 文档先行流程

文档先行不是“先写点说明再做事”，而是先把会影响协作和实现的系统事实写清楚，再进入代码实现。

## 适用范围

以下变更都必须走文档先行：

- API、错误码、鉴权、权限码
- 数据库结构、迁移、事务边界、幂等规则
- 页面行为、路由语义、状态语义、菜单来源
- 配置结构、运行依赖、部署方式
- 业务流程、阶段范围、模块开放边界

以下变更通常不需要触发文档先行确认门禁：

- 纯视觉优化
- 纯布局与样式调整
- 不影响流程的静态文案微调

## 标准步骤

1. 识别这次改动是否会改变契约、行为或长期结构。
2. 找到对应文档或机器契约。
3. 先更新文档、迁移或配置示例。
4. 由维护者确认设计、契约或验收口径。
5. 再开始实现代码。
6. 用最小但有意义的验证命令收尾。

## 文件归属

- AI 工作流和门禁：`AGENTS.md`、`.codex/skills/`
- API 契约：`docs/server/api/`
- 数据库可执行契约：`server/migrations/`
- 数据库设计：`docs/server/database/design.md`
- 前端行为与页面范围：`docs/admin/`、`docs/web/`
- 配置契约：`server/config.example.yaml`
- 部署与开发流程：`docs/development/`、`docs/operations/`
- 计划与进度：`docs/analysis/`、`docs/plan/`、`docs/progress/`

## 进度文档生命周期

`docs/progress/` 是阶段账本，不是最终契约。

功能或阶段完成后，不要求直接删除进度文档，但必须完成两件事：

1. 把长期有效事实沉淀到对应权威文档或机器契约。
2. 让进度文档保持为当前状态记录、历史说明或归档记录。

如果旧进度文档已经不再指导当前实现，应移动到 `docs/progress/archive/`，或在文件顶部标注“历史记录，以当前权威文档为准”。

读取进度文档时，只能用它理解阶段背景和验收历史；真正决定接口、页面、数据库、配置和权限时，必须回到对应权威文档。

## 判断准则

如果一条信息会影响以下任意一方，就不应只存在于 skill 中：

- 维护者的设计判断
- 代码实现
- 接口对接
- 数据迁移
- 测试验收
- 后续迭代

这类信息应该进入文档或机器契约。
