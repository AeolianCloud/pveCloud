# Contract Quality Gate

本文件定义 AI 写 owner docs 或机器契约时的质量门禁。
它不承载具体业务字段、状态值、错误码、表结构或页面契约。

## 适用范围

当任务涉及以下任一内容时，更新 owner docs 或机器契约后，必须先按本门禁自检，再停下来等待维护者确认：

- 状态流转、审批、发布、取消、关闭、释放、确认、同步、刷新、吊销
- 会产生业务事实、资金变化、资源交付、身份校验、通知或异步任务的流程
- 外部平台、回调、Webhook、SMTP、对象存储、MCP/PVE、短信、OAuth
- 资源分配、编号分配、库存、额度、唯一性、幂等键、任务键
- 跨 admin/web/server/database/operations 的联动功能
- 安全、权限、审计、敏感信息、限流、事务或补偿

## 必填问题

如果 owner docs 中没有明确回答下列问题，不能进入实现。
若某项不适用，文档或确认点中必须说明“不适用”的原因。

### 状态机

- 允许的当前状态是什么？
- 成功后的目标状态是什么？
- 非法状态返回什么语义？
- 是否存在中间态、终态或人工处理态？
- 状态变更是否会影响关联对象的状态？

### 并发与锁

- 哪些记录需要在同一事务内锁定？
- 锁粒度是什么：用户、业务对象、任务、资源映射、配置项还是其它业务键？
- 是否需要数据库唯一约束、条件唯一约束、生成列、任务键或短锁兜底？
- 旧表单、重复点击、多 worker、多管理员并发时，哪个写入胜出？
- 是否允许并发读到中间态，前端应该展示什么？

### 幂等与重复提交

- 幂等键来源是什么？
- 同一幂等键重复提交返回已有结果、`409xx` 冲突，还是创建新的业务事实？
- 没有幂等键的写接口如何拒绝或限制重复提交？
- 失败任务重试、外部回调重放、用户刷新页面、管理员重复点击如何处理？

### 外部副作用

- 本地数据库事务内做哪些事？
- 外部调用在事务前、事务后，还是由异步任务执行？
- 外部调用成功但本地更新失败时如何恢复？
- 本地提交成功但外部调用失败时可见状态是什么？
- 外部返回异步 operation 时如何查询、重试、超时和人工处理？
- 哪些上游响应可以保存，哪些只能保存摘要或错误码？

### 异步任务和补偿

- 任务类型、对象、幂等键、计划时间、最大尝试次数和重试延迟由谁决定？
- 任务成功、失败、跳过、取消分别如何影响业务状态？
- 任务达到最大重试次数后，谁能看到、谁能重试、重试是否复用原任务行？
- 旧任务遇到业务状态已变化时是取消、跳过还是失败？

### 权限、审计和敏感信息

- 谁可以查看、创建、修改、取消、重试或释放？
- 前端权限只做可见性还是会影响后端裁决？
- 哪些操作必须写后台审计或用户业务日志？
- 审计、日志、任务 payload/result、通知记录和响应中哪些字段必须脱敏、摘要或禁止保存？

### 测试口径

- 至少需要哪一层长期回归测试：handler、service、repository、integration adapter、migration？
- 必须覆盖哪些失败、重复提交、非法状态、低权限、跨资源、并发或外部不可用分支？
- 如果暂不补测试，剩余风险是什么，是否需要维护者明确接受？

## 输出要求

契约/行为变更在暂停确认前，除常规任务分类外，还要输出：

```text
契约质量门禁：
- 状态机：
- 并发与锁：
- 幂等与重复提交：
- 外部副作用：
- 异步任务/补偿：
- 权限、审计和敏感信息：
- 测试口径：
- 未覆盖项及原因：
```

## 禁止用语

以下表达不能单独作为契约：

- “需要幂等”
- “需要可恢复”
- “注意并发”
- “后续人工处理”
- “失败后可重试”
- “按状态机处理”

必须写清具体状态、锁、键、返回语义、恢复路径和验收方式。
