# pveCloud Workflow Guardrails

本文档只定义 AI 在仓库中的工作方法，不定义项目契约。

## 核心职责

- `CLAUDE.md`：仓库级读取顺序、边界和总规则
- `.claude/skills/pvecloud-document-first.md`：文档先行门禁与执行流程
- `.claude/skills/pvecloud-*.md`：分领域实现守则
- `docs/`：项目事实、契约、架构、计划、进度

## 标准流程

1. 先看 `git status --short`，确认工作区是否有用户未提交改动。
2. 先读入口与对应 skill，不直接跳进代码。
3. 找到本次任务对应的项目文档或机器契约，并在动实现前明确这些 owner docs 已读。
4. 如果刚拉取代码、工作区已有改动，或用户说继续未完成任务，先做代码/文档漂移检查。
5. 先判断是契约/行为变更、纯视觉优化、AI workflow，还是仍需方案澄清。
6. 新功能、复杂业务流、复杂 UI 或跨面方向不清时，先澄清目标、当前依据、影响面、非目标、风险和确认点。
7. 契约/行为变更先更新文档或机器契约，再暂停等待确认。
8. 纯视觉优化可直接实现，但仍要遵守前端 guardrail。
9. AI workflow 变更可直接改 AI 工作流文件，但不能写入或覆盖项目契约。
10. 只在与任务直接相关的目录内改动，避免顺手扩散。
11. 非平凡功能、修复或阶段验收的最终回复，默认补充“后续建议”，给出 1 到 3 条更优技术路线、测试、观测性、安全、性能或维护性建议。
12. 用最小但有意义的验证命令收尾。

## 依赖优先原则

- 进入实现前，先判断是否已有与当前契约匹配的官方 SDK、维护良好的第三方库或仓库内成熟封装可复用。
- 涉及第三方平台、签名验签、对象存储、短信、OAuth、图表、上传、富文本、日期处理或复杂交互时，默认优先复用，不先自研协议层或基础组件。
- 只有在没有合适依赖、依赖不满足安全、许可证、兼容性或维护要求，或维护者明确要求自研时，才自行实现，并在最终说明里交代原因。

## 代码注释策略

新增或修改非平凡代码时，默认补充高价值详细注释。注释要解释业务意图、约束和风险，不机械复述代码。

- 状态机、事务边界、幂等键、权限裁决、安全校验、外部副作用、失败恢复和跨模块联动必须写清“为什么这样做”。
- 复杂函数按关键步骤添加行前或块级注释；简单赋值、普通字段映射、显而易见的 if/return 和常规 CRUD 不加机械注释。
- 维护者明确要求“逐行注释”时，只对指定文件、函数或关键流程执行逐行注释模式。
- 修改已带注释的代码时，同步更新或删除过期注释，不能留下与实现不一致的说明。

## 功能收尾建议

最终回复中的建议必须和已完成内容分开。建议只写可执行的高价值项，例如：

- 是否有更合适的成熟库、官方 SDK 或已有仓库封装可复用。
- 是否需要补更细的 service、handler、repository、前端交互或端到端回归。
- 是否应增强观测性、审计、安全、性能或运维可恢复性。
- 没有实质建议时可写“暂无额外建议”或省略；用户明确要求极简输出时跳过。

## 定向读取

项目变大后，不要把整个 `docs/` 一次性读进上下文。

- 如果目录下有 `README.md`，先读这个索引。
- 再按任务只读对应页面、接口、数据库、配置或进度文档。
- 改 admin 页面时，先读 `docs/admin/README.md`，再读相关 `pages/*.md` 和 `routing-permissions.md`。
- 改 API 时，只读相关接口文档和必要的 conventions。
- 改数据库时，只读相关数据库设计段落和迁移文件。
- 改进度或阶段边界时，才读 `docs/progress/` 的对应文件。
- 如果一次任务需要读取太多不相关文档，先拆任务或暂停确认范围。

## 本地环境探测

AI 工具进程可能不会加载维护者的交互式 shell 初始化脚本。

- 如果普通命令找不到 `node`、`bun`、`go`、`nvm` 等本地版本管理器提供的工具，先用 `bash -ic 'command -v ...; ... --version'` 复核。
- 不要只根据非交互 shell 的 PATH 判定项目缺少运行环境。
- 如果交互 shell 能找到工具，后续构建、测试和开发命令优先用同样的 `bash -ic` 包装。
- `rg` 是首选代码搜索工具；如果不可用，再退回 `git grep`、`grep` 或 `find`。

## 联动判断

当任务不是纯 `admin` 视觉/UI 优化时，先判断是否属于联动开发：

- 用户侧业务流默认联动 `web + server/api + database`
- 运营侧业务流默认联动 `admin + server/api + database`
- 用户和运营共享同一业务事实时默认联动 `web + admin + server/api + database`
- 影响配置、部署、启动脚本、代理边界或依赖顺序时再联动 `operations`

联动开发的默认顺序：

1. 明确受影响的 owner docs 和机器契约
2. 先更新文档/契约
3. 对状态、事务、幂等、外部副作用、异步任务、权限、审计和测试口径执行契约质量门禁
4. 再停下来等待维护者确认
5. 确认后按最小可验证面推进实现

禁止事项：

- 不要只实现其中一个面，然后把其余面当成后续补丁
- 不要先写代码再补契约
- 不要把单面临时可运行当成完整交付

## 漂移检查

文档先行不是只约束"从零开始写代码"的场景，也约束"代码已经先变了"的接手场景。

触发条件：

- 刚执行过 `git pull`
- `git status` 显示已有用户或上游改动
- 用户要求"继续未完成任务"
- 新功能代码已经出现，但文档仍停留在旧阶段、旧页面或旧接口口径

检查方法：

- 用 `git status`、`git diff`、`git log` 或定向读文件确认代码实际变化。
- 把变化映射到权威项目文档或机器契约，例如 `docs/admin/`、`docs/server/api/`、`server/migrations/`、`server/config.example.yaml`。
- 同步检查 `docs/analysis/`、`docs/plan/`、`docs/progress/`，避免计划和进度仍描述旧范围。
- 如果同一事实在多个文档里说法不一致，先按 `CLAUDE.md` 的权威来源判断，再修正文档之间的口径。
- 如果代码和文档已经不匹配，先回填或修正文档/契约，暂停请维护者确认，再继续实现。
- 如果不匹配来自旧进度文档，先按权威契约判断，再更新、归档或标注旧进度文档。

常见原因：

- AI 只读了任务附近的文档，没有横向检查计划、进度和架构文档。
- 上游或用户已经推进了新功能，但没有在同一次变更中同步所有文档。
- skill 只要求"实现前先写文档"，没有要求"接手已有代码时先审计文档是否漂移"。
- 进度文档被当作静态记录，实际功能范围变化后没有被纳入验收同步。

## 契约质量门禁

文档先行不是只要求“先写文档”，还要求 owner docs 写到可以实现、可以评审、可以测试。

触发条件：

- 状态变更、工作流、审批、取消、关闭、释放、确认、同步或刷新
- 外部调用、回调、异步任务、通知、对象存储、MCP/PVE、身份校验或邮件
- 资源分配、库存、额度、编号、VMID、任务键、幂等键或唯一性
- 权限、审计、敏感信息、限流、事务、补偿或失败恢复

处理要求：

1. 读取 `.claude/skills/pvecloud-contract-quality.md`。
2. 在 owner docs、迁移或配置示例中写清具体业务契约。
3. 停确认前输出契约质量门禁摘要。
4. 如果某项暂不适用，说明原因；不能用“注意幂等”“可恢复”“按状态机处理”等笼统表述替代契约。

如果文档没有回答锁粒度、幂等键、非法状态、重复提交、外部失败恢复、审计和测试口径，不进入实现。

## 文档同步回扫

当任务属于仓库级契约调整，尤其是目录结构、架构分层、路由边界、权限层次、配置位置、部署入口或跨模块规则变化时，更新 owner docs 之后不能立刻结束。

必须追加执行一次文档同步回扫：

1. 先更新 owner docs 或机器契约。
2. 再全仓检索 `docs/` 和 `.claude/skills/` 中对旧口径、旧路径、旧命名、旧边界的引用。
3. 把索引文档、引用文档、skill guardrail、workflow 指引一起同步完。
4. 确认不存在会继续误导实现的旧位置说明后，才可以停下来等待维护者确认。

回扫目标至少包括：

- 目录索引文档，例如各级 `README.md`
- API、数据库、架构、部署、进度相关引用文档
- `.claude/skills/` 中引用这些结构的 guardrail 和 workflow

不要只改主文档就结束，也不要把"后续再补外围引用"当成默认流程。

## 进度文档使用规则

`docs/progress/` 是阶段账本，用来恢复背景、验收状态和范围变化历史。

- 读进度文档时，先把它视为历史/状态资料，而不是最终契约。
- 写完功能或阶段后，长期有效事实必须进入 `docs/admin/`、`docs/server/api/`、`docs/server/`、迁移或配置示例等权威位置。
- 阶段完成后，进度文档可以保留，但应减少当前契约细节，只保留状态、结论、时间点和验收结果。
- 如果旧进度文件容易误导后续实现，应移动到 `docs/progress/archive/`，或在顶部标注历史状态。
- 当进度文档和权威文档冲突时，不继续写代码；先同步文档口径并等待确认。

## AI 与文档分工

- Skill 写"AI 应该怎么做"。
- 文档写"系统是什么、接口是什么、页面应该怎样工作"。
- 迁移和配置示例写"机器最终执行什么"。

## 写作原则

- skill 用来减少 AI 重复犯错，不用来隐藏项目事实。
- 文档用来给人和代码共同消费，不用来写提示词。
- 如果某条规则只对 AI 有意义，写进 skill。
- 如果某条规则会影响实现、验收、协作或系统行为，写进文档或机器契约。

## 实现中新增行为的回退门禁

实现过程中经常会发现需要补强安全、校验、事务、幂等、限流、审计、存储或配置。只要这些补强会影响系统接受什么、拒绝什么、保存什么、返回什么、记录什么、如何授权或如何处理敏感数据，就不是纯代码细节。

处理要求：

1. 立即暂停继续写实现。
2. 找到对应 owner docs 或机器契约。
3. 先更新文档或机器契约。
4. 停下来请维护者确认。
5. 确认后再继续实现或调整代码。

不要把“安全增强”“顺手优化”“内部一致性修复”当成绕过文档先行的理由。

## Skill 质量自检

修改 `CLAUDE.md` 或 `.claude/skills/*.md` 时，执行文本自检即可，不要求脚本校验。

- 每个 skill 的 YAML front matter 至少保留 `name` 和 `description`。
- skill 文件名、front matter `name` 和引用用的 `$skill-name` 保持一致。
- skill description 必须说明触发场景，body 必须说明何时使用、何时跳过、输入或验证要求。
- skill 只写 AI 工作流、守则和实现习惯；不得写 API 字段、响应结构、表结构、配置值、页面契约或产品开放状态。
- skill 规则不得和 `CLAUDE.md`、owner docs、迁移或配置示例冲突；冲突时修 skill，不改项目契约来迎合 skill。

详细检查使用 `$pvecloud-skill-quality`。

## Git 提交信息规则

维护者要求提交时，提交信息遵循 Conventional Commits 1.0.0（https://www.conventionalcommits.org/en/v1.0.0/）。说明内容默认使用中文，不能只写一句很短的 subject 就结束。

- 未经维护者在当前对话中明确要求，不执行 `git add`、`git commit`、`git push`，也不主动改变暂存区或历史。

提交信息需要满足：

- subject 格式为 `<type>[optional scope][!]: <description>`；常用 type 包括 `feat`、`fix`、`docs`、`refactor`、`test`、`chore`、`build`、`ci`、`perf`、`style`、`revert`。
- `feat` 用于新能力，`fix` 用于缺陷修复，破坏性变更必须使用 `!` 或在 footer 写 `BREAKING CHANGE:`。
- description 可以使用中文，但必须简短说明本次变更主题，例如 `docs(workflow): 统一 AI 提交规范`。
- 非平凡变更必须有 body；body 默认用中文详细说明为什么需要改、改了哪些方面、如何验证、还有什么风险。文档结构、skill、API、权限、路由、迁移、配置和业务行为变更都属于非平凡变更。
- body 按主题组织，例如背景、主要变更、验证、风险。
- 需要引用 issue、PR、评审人或破坏性变更时，使用 footer；破坏性变更 footer 必须写 `BREAKING CHANGE: <说明>`。
- 如果没有运行测试或构建，要明确说明原因。
- 如果提交已经与 `origin/*` 对齐，不擅自 amend 改写远端历史。
- 如果已发布提交信息太短，新增规则提交修正流程，后续提交按新规则执行。
