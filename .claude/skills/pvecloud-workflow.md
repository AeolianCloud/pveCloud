# pveCloud Workflow Guardrails

## 标准流程

1. 先看 `git status --short`，确认工作区是否有用户未提交改动。
2. 先读入口与对应 skill，不直接跳进代码。
3. 如果刚拉取代码、工作区已有改动，或用户说继续未完成任务，先做代码/文档漂移检查。
4. 找到本次任务对应的项目文档或机器契约。
5. 先判断是契约/行为变更，还是纯视觉优化。
6. 契约/行为变更先更新文档或机器契约，再暂停等待确认。
7. 纯视觉优化可直接实现，但仍要遵守前端 guardrail。
8. 只在与任务直接相关的目录内改动，避免顺手扩散。
9. 用最小但有意义的验证命令收尾。

## 定向读取

项目变大后，不要把整个 `docs/` 一次性读进上下文。

- 如果目录下有 `README.md`，先读这个索引。
- 再按任务只读对应页面、接口、数据库、配置或进度文档。
- 改 admin 页面时，先读 `docs/admin/README.md`，再读相关 `pages/*.md` 和 `routing-permissions.md`。
- 改 API 时，只读相关接口文档和必要的 conventions。
- 改数据库时，只读相关数据库设计段落和迁移文件。
- 改进度或阶段边界时，才读 `docs/progress/` 的对应文件。
- 如果一次任务需要读取太多不相关文档，先拆任务或暂停确认范围。

## 漂移检查

文档先行不是只约束"从零开始写代码"的场景，也约束"代码已经先变了"的接手场景。

触发条件：

- 刚执行过 `git pull`
- `git status` 显示已有用户或上游改动
- 用户要求"继续未完成任务"
- 新功能代码已经出现，但文档仍停留在旧阶段、旧页面或旧接口口径

检查方法：

- 用 `git status`、`git diff`、`git log` 或定向读文件确认代码实际变化。
- 把变化映射到权威项目文档或机器契约，例如 `docs/admin/`、`docs/server/api/`、`server/migrations/`、`server/config.example.yaml`。
- 同步检查 `docs/analysis/`、`docs/plan/`、`docs/progress/`，避免计划和进度仍描述旧范围。
- 如果同一事实在多个文档里说法不一致，先按 `CLAUDE.md` 的权威来源判断，再修正文档之间的口径。
- 如果代码和文档已经不匹配，先回填或修正文档/契约，暂停请维护者确认，再继续实现。
- 如果不匹配来自旧进度文档，先按权威契约判断，再更新、归档或标注旧进度文档。

常见原因：

- AI 只读了任务附近的文档，没有横向检查计划、进度和架构文档。
- 上游或用户已经推进了新功能，但没有在同一次变更中同步所有文档。
- skill 只要求"实现前先写文档"，没有要求"接手已有代码时先审计文档是否漂移"。
- 进度文档被当作静态记录，实际功能范围变化后没有被纳入验收同步。

## 进度文档使用规则

`docs/progress/` 是阶段账本，用来恢复背景、验收状态和范围变化历史。

- 读进度文档时，先把它视为历史/状态资料，而不是最终契约。
- 写完功能或阶段后，长期有效事实必须进入 `docs/admin/`、`docs/server/api/`、`docs/server/`、迁移或配置示例等权威位置。
- 阶段完成后，进度文档可以保留，但应减少当前契约细节，只保留状态、结论、时间点和验收结果。
- 如果旧进度文件容易误导后续实现，应移动到 `docs/progress/archive/`，或在顶部标注历史状态。
- 当进度文档和权威文档冲突时，不继续写代码；先同步文档口径并等待确认。

## AI 与文档分工

- Skill 写"AI 应该怎么做"。
- 文档写"系统是什么、接口是什么、页面应该怎样工作"。
- 迁移和配置示例写"机器最终执行什么"。

## 写作原则

- skill 用来减少 AI 重复犯错，不用来隐藏项目事实。
- 文档用来给人和代码共同消费，不用来写提示词。
- 如果某条规则只对 AI 有意义，写进 skill。
- 如果某条规则会影响实现、验收、协作或系统行为，写进文档或机器契约。
